Безопасность MongoDB

Авторизация в MongoDB обеспечивает запуском демона с параметром --auth. Для начала работы, требуется переключиться на базу данных admin и создать пользователя с правом создавать иных пользователей. При этом, чтобы создать пользователя в нужной базе данных, нужно сначала переключиться на соответствующую базу данных. Авторизацию можно настроить и с помощью mongod.conf файла. Для этого требуется указать параметр при запуске демона Mongo --config. Также стоит включить флаг --noscripting, отключающий выполнение JavaScript скриптов.

Также можно подключить TSL/SSL шифрование при подключении к MongoDB. Только SSL пользователи смогут подключаться к Mongo. И еще момент, что для того, чтобы к mongo можно было подключиться, она должна "открыть" ip-адрес с помощью команды --bind-ip. Обычно, это делается только для приложений, которые обращаются к БД.

Плюсы подобной авторизации:

  1. Все довольно просто настраивается. Стоит запомнить пару команд db.createUser(...) и db.updateUser(...)
  2. Можно создать конфиг файл, с помощью которого проще разворачивать экземпляр БД.
  3. Встроенные инструменты, по типу SSL и bind_ip очень полезны и просты.
Про минусы сложно что-либо сказать. Хоть новостей про утечку данных нагуглил достаточно, но в основном это был человеческих фактор.

Шардинг

Шардинг -- способ представления набора данных, которые хранятся на разных машинах (необязательно в одном кластере), и при этом будут вести себя, как будто они находятся на одной машине. Разница между партициями в том, что партицированные данные находятся на одной машине, когда как шардинговые данные -- на разных. Например, самые частые запросы могут быть направлены к наиболее мощной машине-шарду.

Шард разделяет данные на чанки -- логика близка партицированию: также используется ключ, по которому будут "разделяться" данные по шардам. За поиск данных в шардах отвечает процесс mongos -- некий "маршрутизатор", который знает, где лежат какие данные. Но, если ключ будет установлен неоптимальный, то mongos будет проверять все шарды на данные, что скажется на производительности запросов.

Из лекций по highload узнал, что к шардингу желательно прибегать как можно позже, но не в последний момент. Лучше оптимизировать запросы, индексы, структуру документов, прежде чем переходить к шардингу. Шардинг сложнее разворачивать, поддерживать (devOps'ы не скажут вам спасибо), у них множество нюансов при работе, в особенности с индексами. Обычно команды пишут внутренние инструменты для шардинга, чтобы перенсти данные на новый шард.

Плюсы шардинга:

  1. Можно оптимизировать запрос, путем выставление на самую мощную машину первичный шард
  2. Если нужно в рантайме обновить конфигурации машины, то тут снова поможет шардинг
Минусы:
  1. Тяжело настраивается
  2. Тяжело поддерживается
  3. Тяжело разворачивать
  4. Стоит прибегать только тогда, когда обычные методы оптимизации не помогают.
