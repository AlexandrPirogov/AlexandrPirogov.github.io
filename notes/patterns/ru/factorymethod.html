Factory Method

Что предоставляет?

Предоставляет интрфейс для создания объектов, но позволяет субклассам определять, объект какого класса должен быть создан.

Применимость

Применяйте данный паттерн когда:
  1. класс не может предугадать, какой конкретно класс объекта он должен создать;
  2. класс хочет, чтобы его субклассы опредляли, какой объект создавать;
  3. классы делегируют ответственность одному из нескольких вспомогательных подклассов, и вы хотите локализовать информацию о том, какой вспомогательный подкласс является делегатом.

Мотивация

Представим, что мы создаем приложение по продаже автомобилей. Изначально, мы начали с продажи единственных Жигулей :) По мере роста популярности системы, пользователи хотели бы видеть более широкий ассортимент: различные марки автомобилей, а также возможность приобрести другой вид транспорта (скажем, яхты).

Одна из проблем, с которой мы сталкиваемся, что заранее не знаем, какой конкретно класс создавать, пока пользователь не выберет подходящий ему транспорт. Создание десятков if else снижает гибкость нашего кода и повышает цикломатическую сложность.

Абстрактный класс Транспорт:
class Vehicle
{
public :
    virtual void toString() = 0;
};
Класс "Жигули"
class Jiga : public Vehicle
{
public:
    virtual void toString() { some realization };
};
Класс Лодка
class Yacht : public Vehicle
{
public:
    virtual void toString() { some realization };
};

Фабричный метод позволяет решить данную проблему.

  1. Инкапсулирует код выбора класса, объект которого нужно создать.
  2. Делегирует субклассам, какой объект нужно создать.

Схема

Ниже представлена схема паттерна Factory Method:

  1. AbstractProduct -- абстрактный класс/интерфейс определенного продукта. От него наследуются конкретные продукты.
  2. AbstractCreator -- объявляет фабричный метод. Метод может быть пустым или абстрактным. Подклассы определяют, какой именно объект будут "создавать".
  3. SomeCreatorClient -- класс-клиент, который владеет AbstractCreator'ом.
  4. Client -- клиент, который будет обращаться SomeCreatorClient.

Последствия применения паттерна

Преимущества

  1. Инкапсуляция кода создания объектов.
  2. На протяжении время выполнения программы, мы работает не с реализацией, а с определенным типом данных.

Недостатки

  1. При расширении фабричного метода, придется определять метода в подклассах.
  2. Остается наличие if else.
